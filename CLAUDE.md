## The Competition

Leon picks a pre-built coding agent (OpenHands chosen), customizes it through five levers (Jinja2 prompt, skills files, model selection via OpenRouter, MCP servers, agent config), and everything flows through `arena.yaml`. The agent runs against 246 questions from OfficeQA, which tests grounded reasoning over 89,000 pages of U.S. Treasury Bulletin documents (1939-2025). Scoring uses 1% fuzzy numeric tolerance. Best published score is ~67.8% (Claude Opus 4.5 with Databricks-parsed docs). Without parsed docs, agents top out ~43%.

## Core Philosophy (The BOTCOIN Thesis)

Every decision follows this hierarchy:

1. **The LLM extracts, code computes, everything gets validated.** The agent should never do arithmetic in natural language. It writes Python for every calculation, no exceptions.
2. **Parsing quality is the #1 variable.** The 30+ percentage point gap between raw PDFs and parsed docs is the single largest performance delta. Getting pre-parsed TXT files into the agent's environment is priority zero.
3. **Detect uncertainty, don't proceed through it.** When extraction is ambiguous, the agent should try again rather than guess. "Missing data = wrong answer" from BOTCOIN applies here.
4. **The self-improvement loop is the methodology, not just a tactic.** Every iteration: run → diagnose failures at the sub-pattern level → prioritize by expected value → fix → track regressions.

## Four-Phase Pipeline (Encode in Every Prompt)

The agent must follow these phases in order for every question:

- **Phase 1 — Retrieve:** Parse the question for time period and topic. Use filename conventions (treasury_bulletin_YYYY_MM.txt) to narrow search. Check both the target year and the following year's bulletin.
- **Phase 2 — Extract:** Open documents, locate relevant tables, extract values exactly as printed. Store raw strings with unit indicators. Do NOT convert, round, or interpret at this stage.
- **Phase 3 — Compute:** Write Python for any required calculation. Apply unit conversions explicitly in code. Use scipy/numpy for statistical operations (regression, t-test). Never compute these by hand.
- **Phase 4 — Validate:** Verify units match the question, answer is within plausible range, fiscal/calendar year is handled correctly, and no extracted values are null. Recompute derived values from raw numbers as cross-check.

## Five Known Failure Modes

1. **Table parsing errors** — hierarchical headers cause column shifts, values attributed to wrong headers
2. **Document versioning** — multiple valid values exist depending on publication date; prefer most recent revision
3. **Fiscal year vs calendar year** — U.S. fiscal year starts October (since 1976, July before that); verify question's time period matches table's
4. **Unit confusion** — values appear in millions, thousands, raw dollars; unit indicators can be in title, header, column, or footnote
5. **Visual comprehension** — ~7 questions reference charts; all agents currently fail; deprioritize unless we've squeezed everything else

## Performance Ceiling and Budget Reality

**Theoretical ceiling:** ~70% with oracle-parsed pages (Databricks result). ~13% of questions (~32) require web search and are unanswerable from the corpus alone. Realistic ceiling without web access: ~57%.

**Budget ($100 OpenRouter) is extremely tight.** Estimated cost per full 246-question run:
- GPT-4.1-mini: ~$12/run (cheapest, ~8 full runs possible)
- Claude Haiku 4.5: ~$20/run (~5 full runs possible)
- Claude Sonnet 4.5: ~$100/run (1 run consumes entire budget)
- Claude Opus 4.6: ~$370/run (impossible within budget)

These are rough estimates — validate Day 1 with a single-question test. The key implication: **you cannot iterate with mid-tier or frontier models.** Use GPT-4.1-mini or Haiku for all iteration runs. Reserve any remaining budget for 1 final run on the best affordable model.
